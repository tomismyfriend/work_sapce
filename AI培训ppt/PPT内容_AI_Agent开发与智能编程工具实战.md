## Slide 3 — 为什么现在要关注 AI Agent？

### 从"对话式 AI"到"自主行动 AI"

```
2023    ChatGPT 引爆大模型 → 人人都在"和 AI 聊天"
2024    Copilot / Cursor → AI 深入编程工作流
2025-26 AI Agent → AI 不再只是回答问题，而是自主完成任务
```

### 关键趋势

- 大模型能力持续增强（推理、工具调用、长上下文）
- 基础设施成熟（Function Calling、MCP 协议、向量数据库）
- 企业需求爆发（自动化运维、智能客服、代码生成）

### 一个直观对比

```
传统 AI 应用:
  你: "帮我查一下北京明天的天气"
  AI: "你可以打开天气App查看"  ← 只能告诉你方法

AI Agent:
  你: "帮我查一下北京明天的天气，如果下雨就提醒我带伞"
  Agent: 调用天气API → 发现是雨天 → "北京明天小雨18°C，记得带伞！"
         ← 直接帮你完成
```

> **一句话：AI Agent 是大模型从"玩具"到"工具"的关键跳板。**

## Slide 4 — 什么是 AI Agent？

### 定义

> AI Agent = 大模型（LLM）+ 感知 + 行动 + 记忆 + 规划 + 循环执行

一个能够**感知环境、自主决策、调用工具、完成任务**的智能程序。

### 一个类比

```
LLM   = 一个极其聪明的大脑（能思考、能推理，但被关在房间里）
Agent = 大脑 + 身体（有手有脚、有记忆、能感知、能执行动作）
```

### Agent 与普通 LLM 应用的区别

| 维度 | 普通 LLM 应用 | AI Agent |
|------|-------------|----------|
| 交互方式 | 单轮问答 | 多轮自主执行 |
| 能力边界 | 生成文本 | 调用工具、执行操作 |
| 决策能力 | 被动响应 | 主动规划、拆解任务 |
| 记忆 | 无/短期 | 短期 + 长期记忆 |
| 类比 | 被关在房间里的天才 | 有手有脚的智能助手 |

### Agent 的五大核心特征

```
┌─────────────────────────────────────────────┐
│              AI Agent 五大特征               │
│                                              │
│  ① 感知（Perception）→ 主动获取环境信息     │
│  ② 行动（Action）    → 执行真实操作         │
│  ③ 记忆（Memory）    → 短期 + 长期记忆      │
│  ④ 规划（Planning）  → 任务拆解与动态调整   │
│  ⑤ 循环（Loop）      → 持续推进直到完成     │
│                                              │
│  LLM 本身不具备以上任何一项                  │
│  LLM 只提供"决策大脑"，Agent 框架补齐执行力  │
└─────────────────────────────────────────────┘
```

> **Agent 不是"更聪明的 LLM"，而是"LLM + 执行框架"。接下来我们逐个展开。**

| 特征 | 实现模块 | 具体组件示例 |
|------|---------|------------|

| **③ 记忆** | 记忆管理器（Memory） | 消息历史数组（短期）、向量数据库（长期）、工作记忆 |

### 关键理解

```

  感知 → 感知层模块负责"看"和"听"
  行动 → 工具执行器模块负责"做"（MCP 是其中一种标准化工具接入方式）
  记忆 → 记忆管理器模块负责"记"和"回忆"

  循环 → 编排层模块负责"持续推进"

LLM 只负责一件事：决策（根据当前状态，决定下一步做什么）
其他所有能力都由 Agent 框架的模块提供
```

> **Agent = LLM（大脑）+ 5 个功能模块（感知层 + 工具执行器 + 记忆管理器 + 规划器 + 编排层）。接下来我们逐个展开每个模块。**

## Slide 6 — 基础概念：什么是 LLM

### 大语言模型（Large Language Model）

LLM 是基于海量文本数据训练出来的深度学习模型，本质是一个**下一个词预测器**。

```
输入: "今天天气"
模型计算: P("很") = 0.35, P("不") = 0.28, P("真") = 0.15, ...
输出: 选择概率最高的词 → "很"
然后: "今天天气很" → 预测下一个词 → "好"
循环往复，直到生成完整回答
```

### LLM 怎么训练出来的？（三个阶段）

```
阶段1: 预训练（自监督学习）
  数据: 万亿级文本（网页、书籍、代码...）
  方法: 预测下一个词，不需要人工标注
  结果: 获得语言理解 + 世界知识

阶段2: 监督微调（SFT）
  数据: 人工标注的高质量问答对
  方法: 教模型"听懂指令"并给出有帮助的回答
  结果: 从"续写文本"变成"回答问题"

阶段3: 偏好对齐（RLHF / DPO）
  数据: 人类对多个回答的偏好排序
  方法: 奖励好的回答，惩罚差的回答
  结果: 学会遵循复杂指令、承认不确定性、可靠地调用工具
```

### LLM 的三个关键能力

| 能力 | 含义 | 对 Agent 的意义 |
|------|------|----------------|
| **通用推理** | 能理解复杂指令并推理 | 充当 Agent 的决策大脑 |
| **上下文学习** | 通过 Prompt 中的示例学习，无需重新训练 | 动态定义 Agent 的行为模式 |
| **指令遵循** | 经过对齐训练，能精确遵循指令 | 可靠地执行工具调用规范 |

> **LLM 本身只会"生成文本"——它没有手、没有脚、没有记忆。这就是为什么我们需要 Agent。**

## Slide 7 — 基础概念：什么是 Token

### Token 是大模型处理文本的最小单位

计算机不能直接处理文字，必须把文本转成数字。Token 就是这个转换的产物。

```
Token = 词表中一个条目的 ID = 一段文本片段的数字编号

词表就是一个映射表（以 GPT-4o 为例，约 10 万条）:

  Token ID  |  对应的文本片段
  ─────────────────────────
     1      |  the          ← 一个完整的词
     2      |  ing          ← 一个后缀
     3      |  你            ← 一个汉字
   12345    |  hello
     ...    |  ...
```

### 一个字 = 几个 Token？

| 文本类型 | 示例 | Token 数 |
|---------|------|---------|
| 常见汉字 | "你好" | 2 tokens（1字 ≈ 1 token） |
| 英文单词 | "hello" | 1 token |
| 长英文词 | "unbelievable" | 3 tokens（拆成 un + believ + able） |
| 代码 | "def factorial(n):" | 6 tokens |

### 为什么 Token 很重要？

- **API 按 Token 计费**：输入 + 输出分开计价
- **上下文窗口以 Token 计量**：128K tokens ≈ 10万英文 / 6万中文
- **中文比英文贵 1.5-2 倍**：同样的语义，中文消耗更多 Token

### 控制成本的实用技巧

| 策略 | 效果 |
|------|------|
| 选对模型（简单任务用 GPT-4o-mini） | 成本降低 90% |
| 精简 System Prompt（去掉冗余指令） | 每轮对话节省 75% |
| 设置 max_tokens 限制输出长度 | 防止生成冗长内容 |
| 缓存重复请求 | 相同 Prompt 零费用 |
| 使用 Batch API（非实时任务） | 价格降低 50% |

> **Token 是 AI 时代的"计量单位"。选对模型 + 精简输入 + 缓存复用 = 成本降低 50-80%。**

## Slide 8 — 为什么 LLM 不能直接"操作"

### LLM 的本质：一个纯文本函数

```
LLM 就是一个函数:

  f(text_in) → text_out

  输入: 一串 Token ID 数组
  输出: 一串 Token ID 数组

  就这样。没有文件读写，没有网络请求，没有数据库操作。
  它运行在一个完全隔离的"沙箱"里，和外界没有任何交互通道。
```

### 用一个例子说明

```
用户: "帮我把桌面上的 report.docx 发给张三"

LLM 能做的:
  "你可以这样操作：1. 打开 Outlook 2. 新建邮件 3. 添加附件..."
  → 它只能告诉你怎么做，不能帮你做

LLM 做不到的:
  → 读取文件 ✗  → 打开邮件 ✗  → 发送 ✗
```

### 技术层面的五大限制

| 限制 | 原因 | 影响 |
|------|------|------|
| **无 IO 能力** | 模型架构只有矩阵运算，没有系统调用 | 不能读写文件、不能发网络请求 |
| **无状态** | 每次 API 调用独立，不记住上次对话 | 不能追踪"做到哪了" |
| **无执行环境** | 模型运行在 GPU 上，不是操作系统进程 | 不能运行代码、不能调用 API |
| **单次输出** | 一次调用只生成一段文本就结束了 | 不能"做完一步看看结果再做下一步" |
| **知识截止** | 训练数据有截止日期 | 不知道今天发生了什么 |

### 关键认知

```
LLM 不是"不想"帮你操作，而是"不能"——

就像计算器能算 123×456，但不能帮你把结果写进 Excel。
计算器需要 Excel 这个"外壳程序"来调用它、获取结果、写入文件。

LLM 也需要一个"外壳程序"——这个外壳就是 Agent 框架。
```

> **LLM 提供智能，Agent 提供执行力。没有 Agent 框架，LLM 只是一个"纸上谈兵"的天才。**

## Slide 9 — Agent 特征详解①：感知能力（Perception）

> **实现模块：感知层（Perception Layer）** — 负责主动获取环境信息

### LLM vs Agent 的感知差异

```
LLM:  只能看到你输入的那段文字
      → 像一个被关在隔音房间里的人，只能通过纸条交流

Agent: 能主动获取环境信息
      → 像一个有眼睛、耳朵的人，可以主动观察周围
```

### 为什么感知能力如此重要？

```
没有感知的 Agent = 瞎子摸象

  用户: "帮我优化这个项目的性能"

  没有感知的 Agent:
    只能基于用户描述来猜测问题
    → "可能是数据库查询慢？你可以加索引..."
    → 猜测可能完全错误

  有感知的 Agent:
    1. 读取项目代码 → 发现使用了 N+1 查询
    2. 执行性能分析 → 发现 API 响应时间 2.3s
    3. 查看数据库日志 → 发现慢查询
    4. 综合分析 → "发现 3 处 N+1 查询问题，建议添加 eager loading"
    → 基于真实数据给出准确建议
```

│     ├── Shell 命令 → git status, ls, ps, top, df             │

│     ├── 系统资源 → CPU、内存、磁盘使用率                      │

### 感知的工作流程

```
用户请求: "帮我分析一下这个项目的代码质量"

Agent 感知层的执行流程:

Step 1: 文件系统感知
  → 读取 package.json → 了解项目依赖
  → 遍历 src/ 目录 → 了解代码结构
  → 读取 .eslintrc → 了解代码规范配置

Step 2: 系统命令感知
  → 执行 npm run lint → 获取代码规范问题
  → 执行 npm test → 获取测试覆盖率
  → 执行 git log --oneline -10 → 了解最近提交

Step 3: 综合分析
  → 将感知到的所有信息整合
  → 传递给 LLM 进行推理和决策

结果:
  "项目代码质量分析：
   - ESLint 发现 12 个警告，主要是未使用的变量
   - 测试覆盖率 73%，src/utils/ 目录只有 40%
   - 最近 10 次提交都是小改动，代码审查记录良好
   建议：优先补充 utils 目录的测试，修复 ESLint 警告"
```

### 实际例子对比

```
场景 1: "帮我看看这个项目的测试覆盖率"

LLM:
  "你可以运行 pytest --cov 来查看"
  → 只能告诉你方法

Agent:
  直接执行 pytest --cov → 读取输出 → 分析结果
  → "当前覆盖率 73%，src/utils.py 最低只有 40%，建议优先补充"
  → 直接帮你完成

场景 2: "这个 API 为什么响应这么慢？"

LLM:
  "可能的原因有：1. 数据库查询慢 2. 网络延迟 3. 代码逻辑问题..."
  → 只能列举可能性，无法确定真正原因

Agent:
  1. 执行 curl 测试 API → 响应时间 2.3s
  2. 查看数据库慢查询日志 → 发现 3 个慢查询
  3. 分析代码 → 发现 N+1 查询问题
  → "发现性能瓶颈：users 接口存在 N+1 查询，每次请求执行 101 次 SQL。
      建议添加 eager loading，预计可优化到 50ms 以内"
  → 基于真实数据给出准确诊断

场景 3: "帮我监控服务器状态"

LLM:
  "你可以使用 Prometheus + Grafana 搭建监控系统"
  → 只能告诉你方案

Agent:
  1. 执行 top → CPU 使用率 85%
  2. 执行 df -h → 磁盘使用率 92%
  3. 执行 netstat → 发现异常连接
  → "警告：服务器状态异常
      - CPU 使用率 85%，主要是 node 进程
      - 磁盘使用率 92%，建议清理日志
      - 发现 50 个异常连接来自 IP 192.168.1.100
      建议：立即清理磁盘，检查异常连接"
  → 主动发现问题并给出具体建议
```

### 感知能力的关键价值

| 价值 | 说明 |
|------|------|
| **准确性** | 基于真实数据而非猜测，给出准确建议 |
| **主动性** | 可以主动发现问题，而不是被动等待用户描述 |
| **上下文理解** | 获取完整环境信息，理解问题的真实背景 |
| **实时性** | 获取最新数据，而不是依赖过时的训练数据 |

### 感知工作流的两种模式

```
模式一：固定步骤（开发者预定义）
─────────────────────────────────
开发者在代码中预先定义好感知步骤，每次都执行相同的流程。

示例：代码质量分析
  Step 1: 读取 package.json
  Step 2: 执行 lint 检查
  Step 3: 运行测试覆盖率
  Step 4: 查看 git 历史

如何修改：直接修改代码，添加或删除步骤

优点：可预测、稳定
缺点：不够灵活，可能收集无关信息


模式二：动态决策（Agent 自主决定）
─────────────────────────────────
Agent (LLM) 根据用户问题，自主决定需要感知什么信息。

示例：诊断 API 为什么慢
  LLM 思考 → "先测试响应时间" → 调用 curl
  LLM 思考 → "可能是数据库问题" → 查看慢查询日志
  LLM 思考 → "需要看代码" → 读取 API 代码
  LLM 思考 → "信息够了" → 给出诊断

如何修改：通过 Prompt 引导或调整工具定义

优点：灵活、智能，只收集需要的信息
缺点：不可预测，可能遗漏关键信息


最佳实践：混合模式
─────────────────────────────────
阶段 1: 固定的基础步骤（保证不遗漏关键信息）
阶段 2: 动态补充（根据问题决定是否需要更多信息）

大多数生产环境采用这种模式，平衡可预测性和灵活性。
```

> **感知能力让 Agent 从"纸上谈兵"变成"实地考察"。就像一个优秀的医生不会只听患者描述，而是会亲自检查、化验、观察，才能做出准确诊断。**

## Slide 10 — Agent 特征详解②：行动能力（Action）

> **实现模块：工具执行器（Tool Executor）** — 负责执行真实操作。MCP 是其中一种标准化工具接入方式

### LLM vs Agent 的行动差异

```
LLM:  只能输出文本
Agent: 能执行真实操作
```

### Agent 的行动方式

```
├── 文件操作 → 创建、读取、修改、删除文件
├── 代码执行 → 运行 Python / Shell / SQL 命令
├── API 调用 → 发送邮件、创建工单、调用第三方服务
├── 浏览器操作 → 打开网页、点击按钮、填写表单
└── 系统操作 → 安装依赖、启动服务、管理进程
```

### 实际例子

```
用户: "给这个函数加上单元测试"

LLM:
  输出测试代码文本
  → 你需要自己复制粘贴到文件里，自己运行

Agent:
  读取源文件 → 生成测试 → 写入 test_xxx.py → 运行 pytest → 确认通过
  → 全程自动完成
```

### 行动能力的技术实现：Function Calling

```
Agent 的行动能力通过 Function Calling 实现:

  1. 开发者定义工具列表（JSON 格式）
     → 告诉 LLM "你有哪些工具可以用"

  2. LLM 推理后输出结构化的工具调用指令
     → 不是自由文本，而是严格的 JSON

  3. Agent 框架解析指令，执行真正的操作
     → LLM 不执行任何代码，它只是"说"要调用什么

  4. 执行结果返回给 LLM，继续推理
     → 形成闭环
```

> **LLM 只负责"决策"（调用什么工具、传什么参数），Agent 框架负责"执行"。**

## Slide 11 — Agent 特征详解③：记忆系统（Memory）

> **实现模块：记忆管理器（Memory Manager）** — 负责存储和检索上下文信息

### 为什么 LLM 没有记忆？

```
LLM 是一个纯函数：每次调用都是独立的，不保留任何状态

  第 1 次调用: llm("你好") → "你好！有什么可以帮你的？"
  第 2 次调用: llm("我刚才说了什么？") → "你没有说过什么，这是我们的第一次对话"
                                           ↑ LLM 完全不记得第 1 次调用

原因：LLM 运行在 GPU 上，每次调用结束后，中间状态全部丢弃。
      模型参数是固定的（推理时不更新），不会"记住"任何新信息。

Agent 的记忆 = 在 LLM 外部维护的数据结构，每次调用时注入到 Prompt 中。
```

### Agent 的三种记忆类型

```
┌──────────────────────────────────────────────────────────────┐
│                     Agent 记忆系统                            │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  ① 短期记忆（Short-term Memory / Working Memory）    │     │
│  │                                                      │     │
│  │  存储位置: messages 数组（内存中）                     │     │
│  │  生命周期: 当前任务（任务结束即清除）                   │     │
│  │  内容: 完整的对话历史 + 工具调用记录                   │     │
│  │  作用: 让 LLM 知道"我们刚才聊了什么"                  │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  ② 长期记忆（Long-term Memory）                      │     │
│  │                                                      │     │
│  │  存储位置: 向量数据库（持久化存储）                     │     │
│  │  生命周期: 永久（跨任务、跨会话）                       │     │
│  │  内容: 过去的经验、项目文档、用户偏好                   │     │
│  │  作用: 让 Agent 能"回忆"过去的知识和经验               │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  ③ 工作记忆（Scratchpad）                            │     │
│  │                                                      │     │
│  │  存储位置: 变量 / 字典（内存中）                       │     │
│  │  生命周期: 当前任务                                   │     │
│  │  内容: 中间计算结果、任务进度、待办清单                 │     │
│  │  作用: 让 Agent 能追踪"做到哪了、还需要做什么"         │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### ① 短期记忆：实现原理

```
短期记忆 = messages 数组，每次调用 LLM 时完整传入

messages = [
    {"role": "system",    "content": "你是一个智能助手..."},
    {"role": "user",      "content": "帮我查北京天气"},
    {"role": "assistant", "tool_calls": [{"name": "get_weather", "args": {...}}]},
    {"role": "tool",      "content": "北京：小雨，18°C"},
    {"role": "assistant", "content": "北京明天小雨，记得带伞"},
    {"role": "user",      "content": "那上海呢？"},        ← 新消息追加到这里
]

# 每次调用 LLM，都把完整的 messages 传入
response = llm.chat(messages)

# LLM 看到完整历史，知道用户之前问了北京天气
# 所以理解"那上海呢"是在问上海的天气
```

**核心问题：上下文窗口有限**

```
假设上下文窗口 = 128K tokens（约 10 万字）

  对话 10 轮 → 约 5,000 tokens → 没问题
  对话 100 轮 → 约 50,000 tokens → 开始紧张
  对话 500 轮 → 约 250,000 tokens → 超出窗口！

解决方案：摘要压缩

  messages = [
      {"role": "system", "content": "你是一个智能助手..."},
      {"role": "system", "content": "[历史摘要] 用户之前讨论了：1. Python 安装 2. 虚拟环境配置 3. pip 使用"},
      # ↑ 前 50 轮对话被压缩成一段摘要
      {"role": "user", "content": "现在帮我安装 requests 库"},  ← 最近的消息保留完整
      {"role": "assistant", "content": "好的，执行 pip install requests..."},
  ]

压缩策略：
  保留最近 N 轮完整对话 + 更早的对话压缩为摘要
  摘要由 LLM 生成："请用 100 字总结以上对话的关键信息"
```

### ② 长期记忆：实现原理

```
长期记忆 = 向量数据库 + Embedding 模型 + 语义检索

完整流程：

【存储阶段】（当 Agent 完成一个任务后）
  1. 提取关键信息："用户偏好使用 Docker 部署，端口映射 8080:80"
  2. Embedding 模型将文本转为向量：
     "用户偏好使用 Docker 部署" → [0.23, -0.15, 0.87, ...] (1536维)
  3. 存入向量数据库：
     {vector: [0.23, -0.15, 0.87, ...], text: "用户偏好使用 Docker 部署", metadata: {...}}

【检索阶段】（当 Agent 需要回忆时）
  1. 用户说："像上次那样帮我部署"
  2. Embedding 模型将查询转为向量：
     "像上次那样帮我部署" → [0.21, -0.12, 0.85, ...]
  3. 在向量数据库中查找最相似的向量（余弦相似度）：
     查询向量 vs 存储向量 → 找到最相似的记忆
  4. 返回相关记忆文本，注入到 LLM 的 Prompt 中

  → LLM 看到："用户偏好使用 Docker 部署，端口映射 8080:80"
  → LLM 理解用户意图，按偏好执行
```

**向量检索的核心思想**

```
语义相似 = 向量距离近

  "帮我部署到测试环境"    → [0.21, -0.12, 0.85, ...]
  "用户偏好 Docker 部署"  → [0.23, -0.15, 0.87, ...]  ← 向量很近！
  "今天天气很好"          → [0.91, 0.33, -0.42, ...]  ← 向量很远

即使措辞完全不同，只要语义相似，就能检索到：
  "部署项目" ≈ "上线服务" ≈ "发布应用" → 向量都很接近
```

**常用向量数据库**

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| Chroma | 轻量、Python 原生 | 本地开发、原型验证 |
| Pinecone | 全托管云服务 | 生产环境、无需运维 |
| Milvus | 高性能、分布式 | 大规模数据 |
| pgvector | PostgreSQL 插件 | 已有 PG 的团队 |

### ③ 工作记忆：实现原理

```
工作记忆 = Agent 在任务执行过程中维护的变量和状态

示例：一个多步骤任务的工作记忆

scratchpad = {
    "task": "重构用户模块",
    "completed_steps": [
        "✅ 读取现有代码",
        "✅ 提取公共函数到 utils.py",
    ],
    "current_step": "更新 import 语句",
    "pending_steps": [
        "⬜ 运行测试确认通过",
        "⬜ 检查代码规范",
    ],
    "intermediate_results": {
        "extracted_functions": ["validate_email", "hash_password"],
        "affected_files": ["user.py", "auth.py", "utils.py"],
    }
}

# 每次调用 LLM 时，将 scratchpad 注入到 Prompt 中
messages = [
    {"role": "system", "content": f"当前任务进度：{scratchpad}"},
    ...
]

# LLM 看到进度，知道下一步该做什么
# 执行完一步后，更新 scratchpad
```

### 三种记忆的协作

```
场景：用户说"像上次那样帮我部署到测试环境"

① 短期记忆：
   当前对话历史 → 知道用户刚才说了什么

② 长期记忆：
   检索"部署"相关记忆 → 找到上次的部署记录
   "上次用了 docker-compose up -d，端口映射 8080:80"

③ 工作记忆：
   记录当前部署进度
   completed: [检查配置, 构建镜像]
   pending: [启动容器, 验证服务]

三种记忆注入 LLM Prompt:
  System Prompt: [长期记忆检索结果]
  Messages:      [短期记忆 - 完整对话历史]
  Scratchpad:    [工作记忆 - 当前任务进度]

→ LLM 拥有完整上下文，做出准确决策
```

### 记忆系统的生命周期

```
┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  用户输入                                                     │
│      ↓                                                        │
│  ① 检索长期记忆 → 找到相关历史经验                             │
│      ↓                                                        │
│  ② 注入 Prompt（长期记忆 + 短期记忆 + 工作记忆）               │
│      ↓                                                        │
│  ③ LLM 推理 → 决定下一步行动                                  │
│      ↓                                                        │
│  ④ 执行行动 → 获取结果                                        │
│      ↓                                                        │
│  ⑤ 更新短期记忆（追加新消息）                                  │
│  ⑥ 更新工作记忆（更新进度）                                    │
│      ↓                                                        │
│  ⑦ 任务完成后 → 提取关键信息 → 存入长期记忆                   │
│      ↓                                                        │
│  ⑧ 清理短期记忆和工作记忆（为下一个任务准备）                  │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

> **记忆系统是 Agent 从"一次性工具"进化为"持续学习的助手"的关键。短期记忆让 Agent 理解上下文，长期记忆让 Agent 积累经验，工作记忆让 Agent 追踪进度。**

## Slide 12 — Agent 特征详解④：任务规划（Planning）

> **实现模块：规划器（Planner）** — 负责任务拆解和动态调整

### LLM vs Agent 的规划差异

```
LLM:  一次回答一个问题
Agent: 能把复杂任务拆解为多步子任务，按顺序执行
```

### Agent 的规划过程

```
用户: "帮我做一个用户注册功能"

LLM: 一次性输出所有代码（可能遗漏细节）

Agent 的规划:
  Step 1: 看看项目用的什么框架 → 读取 package.json
  Step 2: 了解现有代码结构 → 读取 src/ 目录
  Step 3: 设计数据模型 → 创建 User model
  Step 4: 实现 API 接口 → 创建 controller + routes
  Step 5: 添加数据校验 → 中间件
  Step 6: 写单元测试 → 测试文件
  Step 7: 运行测试确认通过 → pytest
  Step 8: 检查代码规范 → eslint
```

### 常见的规划模式

**ReAct（Reasoning + Acting）**
```
思考 → 行动 → 观察结果 → 再思考 → 再行动 → ...
交替进行推理和行动，逐步逼近目标
```

**Plan-and-Execute**
```
先制定完整计划 → 逐步执行 → 检查结果 → 必要时调整计划
```

### 自我反思（Reflection）

```
Agent 执行后还能自我评估:
  写代码 → 运行测试 → 测试失败 → 分析原因 → 修改代码 → 重新测试 → 通过

这让 Agent 从"一次性生成"进化为"迭代优化"
```

## Slide 13 — Agent 特征详解⑤：循环执行（Agent Loop）

> **实现模块：编排层（Orchestrator）** — 负责持续推进任务直到完成

### 这是 Agent 最核心的特征

```
LLM:  一次调用，一次输出，结束
Agent: 循环执行，直到任务完成
```

### Agent Loop 的核心逻辑

```python
# Agent 的核心逻辑，就这么简单
def run_agent(user_input):
    messages = [system_prompt, user_input]

    while True:
        # 1. 让 LLM 思考下一步
        response = llm.chat(messages, tools=available_tools)

        # 2. 如果 LLM 决定调用工具 → 执行工具，把结果喂回 LLM
        if response.tool_calls:
            for tool_call in response.tool_calls:
                result = execute_tool(tool_call)
                messages.append({"role": "tool", "content": result})
            # 继续循环，让 LLM 根据工具结果决定下一步

        # 3. 如果 LLM 认为任务完成 → 返回最终回答
        else:
            return response.content
```

### 执行过程可视化

```
用户: "帮我查一下北京明天的天气，如果下雨就提醒我带伞"

第1轮: LLM 思考 → "需要先查天气" → 调用 get_weather(北京, 明天)
第2轮: LLM 看到结果(小雨) → "是雨天，需要提醒" → 生成最终回答
第3轮: 无工具调用 → 返回: "北京明天小雨，记得带伞！"
```

### 没有循环 vs 有循环

```
没有循环（普通 LLM 应用）:
  用户提问 → LLM 回答 → 结束
  问题: LLM 不知道天气，只能瞎编或说"我不知道"

有循环（Agent）:
  用户提问 → LLM 思考 → 调用工具 → 获取结果 → 继续思考 → 再调用工具 → ... → 完成
  优势: 可以多次调用工具，逐步完成任务
```

> **Agent 的"自主性"不是魔法，而是 LLM 的推理能力 + 循环执行 + 工具调用的组合。**

## Slide 14 — 一个完整的 Agent 系统：模块协作示例

### 场景：用户请求"帮我查一下北京明天的天气，如果下雨就提醒我带伞"

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 系统执行流程                         │
└─────────────────────────────────────────────────────────────┘

【1. 编排层启动】
    ↓
    接收用户输入，初始化消息历史
    进入 Agent Loop

【2. 规划器工作】
    ↓
    分析任务："需要查天气 → 判断是否下雨 → 条件性提醒"
    制定执行计划：先调用天气工具，再根据结果决定下一步

【3. LLM 决策（第1轮）】
    ↓
    思考："我需要先获取北京明天的天气数据"
    输出：调用 get_weather(city="北京", date="明天")

【4. 工具执行器行动】
    ↓
    解析 LLM 的工具调用指令
    通过 MCP Server 调用天气 API
    返回：{"weather": "小雨", "temp": "18°C"}

【5. 记忆管理器存储】
    ↓
    将工具调用和结果追加到消息历史
    messages = [
      {"role": "user", "content": "帮我查一下北京明天的天气..."},
      {"role": "assistant", "tool_calls": [...]},
      {"role": "tool", "content": "{\"weather\": \"小雨\"...}"}
    ]

【6. LLM 决策（第2轮）】
    ↓
    感知层读取工具结果："小雨，18°C"
    思考："是雨天，需要提醒用户带伞"
    输出：直接回复文本（无工具调用）

【7. 编排层判断完成】
    ↓
    检测到 LLM 没有调用工具
    返回最终回答："北京明天小雨，18°C，记得带伞！"
```

### 模块协作总结

```
用户请求
    ↓
[编排层] 启动 Agent Loop
    ↓
[规划器] 分析任务，制定计划
    ↓
[LLM] 决策：调用天气工具
    ↓
[工具执行器] 执行 get_weather（通过 MCP）
    ↓
[记忆管理器] 存储工具结果到消息历史
    ↓
[LLM] 决策：生成最终回答
    ↓
[编排层] 检测完成，返回结果
    ↓
用户收到回答
```

### 关键理解

```
1. 每个特征都有对应的独立模块实现
   - 感知 → 感知层（读取工具返回的数据）
   - 行动 → 工具执行器（MCP Server 调用天气 API）
   - 记忆 → 记忆管理器（存储消息历史）
   - 规划 → 规划器（分析任务结构）
   - 循环 → 编排层（while 循环控制流程）

2. LLM 只负责"决策"
   - 决定调用什么工具
   - 决定何时完成任务
   - 不执行任何实际操作

3. 模块之间通过消息传递协作
   - 编排层是"总指挥"，协调所有模块
   - 记忆管理器是"共享黑板"，所有模块都能读写
   - 工具执行器是"手脚"，LLM 是"大脑"
```

> **这就是一个完整的 Agent 系统：5 个功能模块 + LLM 大脑，通过编排层协调工作。接下来我们深入每个模块的技术细节。**

## Slide 15 — Agent 如何调用工具：Function Calling 详解

### 完整流程

```
  开发者定义工具列表（JSON Schema）
              ↓
  工具描述注入 Prompt（告诉 LLM 有哪些工具可用）
              ↓
  LLM 收到用户消息 + 工具描述
              ↓
  LLM 推理：这个任务需要外部信息吗？
       ↓                    ↓
      需要                   不需要
       ↓                    ↓
  输出结构化调用指令        直接生成文本回答
  {
    "tool": "search_web",
    "args": {"query": "..."}
  }
       ↓
  Agent 框架解析 → 执行工具 → 结果注入上下文
       ↓
  LLM 收到工具结果 → 继续推理
```

### 关键技术点

1. **工具描述是 Prompt 的一部分** — LLM 通过阅读工具的 name、description、parameters 来理解工具能做什么
2. **LLM 输出的是结构化 JSON** — 不是自由文本，而是严格符合 JSON Schema 的调用指令
3. **执行在 Agent 框架侧** — LLM 不执行任何代码，它只"说"要调用什么，Agent 框架负责真正执行
4. **结果回传形成闭环** — 工具执行结果作为新消息注入，LLM 基于结果继续推理

### 为什么工具描述的质量很重要？

```
❌ 差的工具描述:
  name: "search"
  description: "搜索东西"

✅ 好的工具描述:
  name: "search_web"
  description: "搜索互联网获取实时信息，适用于查询新闻、天气、股价等动态数据"
  parameters: {
    "query": {"type": "string", "description": "搜索关键词，尽量具体"}
  }

LLM 根据描述来决定是否使用、如何使用这个工具。
描述越清晰，LLM 的决策越准确。
```

## Slide 16 — MCP 协议：AI 工具的"USB-C 接口"

### 什么是 MCP？

**MCP（Model Context Protocol）= AI 模型连接外部工具的标准协议**

### 用一个类比理解

```
USB-C 之前:
  iPhone 用 Lightning
  安卓用 Micro-USB
  耳机用 3.5mm
  → 每个设备一种接口，很麻烦

USB-C 之后:
  手机、电脑、耳机、充电器都用同一个接口
  → 一个接口通吃所有设备

MCP 就是 AI 领域的"USB-C":
  之前: 每个 AI 工具都需要写定制集成代码
  之后: 一个标准协议，所有 AI 模型通用
```

### 解决什么问题？

```
没有 MCP 的情况:

  假设你有 3 个 AI 工具（ChatGPT、Claude、Cursor）
  想连接 5 个外部服务（GitHub、数据库、Slack、Jira、文件系统）
  
  需要写: 3 × 5 = 15 套定制代码
  每增加一个 AI 工具或外部服务，都要写新的集成代码
  → 工作量爆炸，不可持续

有了 MCP:

  每个外部服务只需实现一次 MCP Server（5 个）
  每个 AI 工具只需实现一次 MCP Client（3 个）
  
  总共: 5 + 3 = 8 个实现
  → 工作量大幅减少，可持续扩展
```

### 实际使用场景

```
场景: 开发者用 Cursor 开发项目

没有 MCP:
  Cursor 只能操作本地文件 + 执行命令
  想查 Jira 工单 → 手动复制粘贴
  想更新 Slack 状态 → 手动操作

有了 MCP:
  安装 Jira MCP Server → Cursor 直接读取和更新工单
  安装 Slack MCP Server → Cursor 直接发消息
  安装 PostgreSQL MCP Server → Cursor 直接查询数据库
  
  开发者: "帮我查一下 Jira 上标记为 P0 的 bug，分析代码找到原因"
  Cursor Agent:
    → 调用 Jira MCP Server 获取 P0 bug 列表
    → 读取相关代码文件
    → 分析 bug 原因
    → 修改代码
    → 在 Jira 上更新工单状态
    → 全程自动完成
```

### MCP 与 Function Calling 的关系

```
Function Calling: LLM 层面的能力
  → LLM 能"决定"调用什么工具
  → 解决"大脑怎么下指令"

MCP: 工具接入层面的标准
  → 定义了工具怎么被发现、描述、调用
  → 解决"手脚怎么标准化接入"

两者是互补的:
  LLM（Function Calling）→ Agent 框架 → MCP Client → MCP Server → 外部系统
  （大脑决策）            （编排层）     （标准接口）   （工具实现）   （真实世界）
```

### MCP 的核心价值

| 价值 | 说明 |
|------|------|
| **标准化** | 工具厂商只需实现一次，所有 AI 应用都能用 |
| **降低成本** | AI 应用不需要为每个工具写定制代码 |
| **生态加速** | 社区共建 MCP Server，工具数量快速增长 |
| **可插拔** | Agent 可以动态发现和加载工具，不需要硬编码 |

> **MCP 让 AI Agent 从"只能操作本地文件"进化为"能连接整个互联网"。就像 USB-C 让设备互联互通，MCP 让 AI 与外部世界无缝连接。**

## Slide 17 — Agent 怎么知道任务完成了？

### 核心答案：不是代码判断的，是 LLM 自己决定的

```
LLM 每次被调用后，只有两种可能的输出:

输出类型 A: 调用工具（tool_calls 不为空）
  → Agent 框架执行工具 → 结果喂回 LLM → 继续循环

输出类型 B: 直接回复文本（tool_calls 为空）
  → Agent 框架认为任务完成 → 返回给用户
```

### LLM 是怎么"学会"什么时候该停的？

```
在训练阶段（RLHF）学到的行为模式:

  用户问 "你好"
  → LLM 判断：不需要工具，直接回答

  用户问 "北京天气怎么样"
  → LLM 判断：需要调用 get_weather
  → 收到工具结果后，判断：信息够了，可以回答了
  → 输出最终文本回复（= 任务完成信号）
```

### 但是 LLM 可能判断错误！所以需要防护

```python
def run_agent(user_input, max_steps=10):
    for step in range(max_steps):  # ← 防护1: 最大步数限制，防止无限循环
        response = llm.chat(messages, tools=tools)

        if response.tool_calls:
            # 执行工具...
            pass
        else:
            # ← 防护2: 可以加验证步骤
            if not validate(response.content):
                messages.append("你的回答似乎不完整，请继续")
                continue  # 强制 LLM 继续工作
            return response.content
```

> **模型越强，判断越准确；模型越弱，越容易过早停止或无限循环。**

## Slide 18 — Agent 怎么判断结果是否正确？

### 三种验证方式

**方式一：程序化验证（最可靠）**
```
适用于有明确正确答案的场景:
  数学计算 → 用代码验证结果
  代码生成 → 运行测试验证
  API 调用 → 检查 HTTP 状态码
```

**方式二：LLM 自我评估（LLM-as-Judge）**
```
适用于开放式任务（文案、翻译、总结）:
  让 LLM 评估自己的回答:
  "这个回答是否正确完成了任务？有哪些问题？"
  → 发现问题 → 反馈给 LLM 修正
```

**方式三：双模型交叉验证（最严谨）**
```
适用于高风险场景（医疗、法律、金融）:
  模型 A 执行任务 → 模型 B 独立执行同一任务 → 模型 C 比较两个结果
```

### 各验证方式对比

| 验证方式 | 适用场景 | 可靠性 | 成本 |
|---------|---------|--------|------|
| 程序化验证 | 数学、代码、API | 最高 | 最低 |
| LLM 自我评估 | 文案、翻译、总结 | 中等 | 中等 |
| 双模型交叉验证 | 医疗、法律、金融 | 最高 | 最高 |

> **可靠的 Agent 需要在关键节点加入验证，越重要的任务越需要多重验证。**

## Slide 19 — 从零手写 Agent：50行代码实现

### 不用任何框架，直接调用 OpenAI API

```python
import openai, json

client = openai.OpenAI()

# 第1步：定义工具（告诉 LLM 有哪些工具可用）
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名"},
                    "date": {"type": "string", "description": "日期"}
                },
                "required": ["city"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "send_notification",
            "description": "发送通知消息给用户",
            "parameters": {
                "type": "object",
                "properties": {
                    "message": {"type": "string", "description": "通知内容"}
                },
                "required": ["message"]
            }
        }
    }
]

# 第2步：实现工具的实际执行逻辑（这是你写的代码，不是 LLM 执行的）
def execute_tool(name, args):
    if name == "get_weather":
        return json.dumps({"city": args["city"], "weather": "小雨", "temp": "18°C"})
    elif name == "send_notification":
        print(f"[通知已发送] {args['message']}")
        return json.dumps({"status": "success"})
    return json.dumps({"error": "未知工具"})

# 第3步：Agent 主循环（核心！）
def run_agent(user_input, max_steps=10):
    messages = [
        {"role": "system", "content": "你是一个智能助手，可以查天气和发通知。"},
        {"role": "user", "content": user_input}
    ]

    for step in range(max_steps):
        response = client.chat.completions.create(
            model="gpt-4o", messages=messages, tools=tools
        )
        msg = response.choices[0].message

        if msg.tool_calls:
            # LLM 想调用工具 → 执行 → 把结果喂回 LLM
            messages.append(msg)
            for tc in msg.tool_calls:
                result = execute_tool(tc.function.name, json.loads(tc.function.arguments))
                messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
        else:
            # LLM 没有调用工具 → 任务完成
            return msg.content

# 运行
result = run_agent("帮我查一下北京明天天气，如果下雨就提醒我带伞")
print(result)
# 输出: "北京明天小雨，18°C，记得带伞！"
```

### 这段代码揭示了什么？

- **LLM 本身不执行任何操作** — 它只是输出"我想调用 get_weather"
- **你的代码负责真正执行** — execute_tool 是你写的 Python 函数
- **Agent 就是胶水代码** — 把 LLM 的决策和真实工具连接起来的循环

## Slide 20 — Agent 的自我纠错能力

### Agent 如何处理错误？

```
传统程序: 遇到错误 → 崩溃或跳过
Agent:     遇到错误 → 理解错误 → 调整策略 → 重试
```

### 实际案例

```
场景: Agent 尝试运行测试

第1轮: 调用 run_tests() → 返回 "3 tests failed"
第2轮: LLM 分析失败原因 → "test_auth 失败是因为缺少环境变量"
第3轮: 调用 set_env("AUTH_KEY", "test-key") → 成功
第4轮: 再次调用 run_tests() → 返回 "All tests passed" ✓
```

### 为什么 LLM 能"理解"错误并修正？

```
1. 训练数据中包含大量"错误→修正"的模式
   代码 debug 过程、数学证明的纠错、文章修改历史

2. 工具返回的错误信息成为新的上下文
   错误堆栈、测试失败信息 → 都是 LLM 的"反馈信号"

3. LLM 的推理能力让它能"诊断"问题
   不只是盲目重试，而是分析错误原因，针对性修正
```

### 三种反思模式

```
① 结果验证: 执行操作 → 检查结果 → 不符合则修正
② 自我评估: 生成输出 → 评估质量 → 低分则重新生成
③ 多路径探索: 尝试多种解法 → 比较结果 → 选择最优
```

> **自我纠错让 Agent 从"一次性生成"进化为"迭代优化"——这是 Agent 能处理复杂任务的关键能力。**

```
① LLM 是什么
   → 下一个词预测器，能推理、能生成，但只能输出文本

② Token 是什么
   → 词表中的一个条目，一个整数 ID，模型处理文本的最小单位

③ 为什么 LLM 不能直接操作
   → 架构限制：无 IO、无状态、无执行环境、单次输出

④ Agent 是什么
   → LLM + 执行框架，补齐了感知、行动、记忆、规划、循环

⑤ Agent 怎么独立完成任务
   → Agent Loop 循环 + Function Calling + 自我纠错

⑥ 怎么用好 AI 工具
   → 选对模型、提供上下文、精简 Prompt、始终 Review
```

### 一句话总结

> **LLM 提供智能，Agent 提供执行力。Agent 不是更聪明的 LLM，而是给天才装上了身体。**
